# myWiki 摄入

用户给你一份素材（文件、URL、粘贴的文本、口述）。你的工作：从中提炼**理解**，写成结构化的 wiki 节点，并将其连入已有的知识图谱。

**这不是做笔记。你不是在复制、也不是在总结。你是在蒸馏用户的理解——他学到了什么、这意味着什么、它跟已有的知识怎么联系。**

---

## ⚠️ 核心质量标准（每个节点必须通过）

在写任何节点之前，牢记这三条底线：

1. **标题必须表达张力或洞察，不是命名概念**
   - ❌ "Middleware 洋葱模型" — 这只是给概念起了个名
   - ❌ "ReAct 循环是 Agent 的本质" — 这只是一个定义声明
   - ✅ "洋葱模型让能力解耦成为配置问题" — 揭示了因果
   - ✅ "Subagent 用上下文隔离换 token 效率，代价是丢细节" — 表达了 trade-off

2. **正文必须有推理链，不是结论罗列**
   - 节点正文至少 3-5 段（150-400 字），包含"因为…所以…"的逻辑
   - 如果你发现自己在列要点（bullet points），你在做笔记不是在写理解

3. **meta_type 必须诚实**
   - 如果正文只是描述"X 是什么/怎么做"→ 那是 `observation`，不是 `insight`
   - `insight` 的试金石：**删掉标题后，读正文能感受到"哦原来如此"的顿悟感吗？**
   - 从学习笔记中提炼的内容，大部分是 `observation`，少数才能升级为 `insight`

## Step 0 — 定位 wiki

wiki 数据在 `content/` 目录下。通过查找包含 `nodes/`、`raw/`、`meta/` 子目录的目录来定位。

```
content/
├── nodes/          # wiki 节点（平铺，无子目录）
├── raw/            # 原始素材（不可变，只读）
│   └── assets/     # 图片等附件
└── meta/
    ├── index.md    # 节点索引（你维护）
    └── log.md      # 操作日志（你追加）
```

找不到时询问用户："你的 myWiki content 目录在哪？"

## Step 1 — 保存素材

将原始素材保存到 `content/raw/YYYY-MM-DD-<简短标题>.md`。

- URL 素材：先获取内容再保存
- 图片：保存到 `content/raw/assets/`
- **永远不修改 `raw/` 下的已有文件**
- 口述很短（< 3 句）时可跳过保存 raw，直接写节点

## Step 2 — 读取已有上下文（分层搜索）

写任何东西之前，用分层策略了解已有知识，避免重复并为 Step 5 的关系发现做准备。

**第一层：扫索引（必做）**
- 读 `content/meta/index.md`，快速浏览所有节点标题和一句话描述
- 标记出"跟本次素材可能相关"的候选节点（10-15 个）

**第二层：读候选 frontmatter（必做）**
- 读取候选节点的 frontmatter（不读正文），看其 domains、relations、status
- 缩小范围到 5-8 个"最值得深读"的节点

**第三层：精读相关节点（必做）**
- 完整读取这 5-8 个节点的正文
- 这些就是你在 Step 5 建立关系时的"候选关联池"

**为什么要分层**：wiki 有 200+ 节点时不可能全部读完。分层搜索让关系发现质量不随 wiki 规模下降。即使 wiki 很小（< 30 节点），也建议按这个流程走——养成习惯。

> 📝 将候选关联池记在心里（或列在内部笔记中），Step 5 时从这个池子里重点发现关系。

## Step 3 — 提取知识点（严格筛选）

分析素材。**先通读全部素材**，在心里形成整体画面，然后才开始切割知识点。

对每个候选知识点，问三个问题：
1. **这是理解还是信息？** — "Redis Stream 支持消费者组"是信息（跳过）；"用 Redis Stream 做事件总线的核心好处是消费者组天然提供 at-least-once + 可回溯"是理解（保留）
2. **这个理解是独立的吗？** — 如果它只有依附在另一个概念上才有意义，考虑合并而不是单独建节点
3. **半年后重读这个节点，我还能学到东西吗？** — 如果答案是"不能，因为太浅了"，要么写深，要么跳过

**判断结果：**
- **新建节点** — 真正新的理解/模型/判断/疑问，且能写出 3 段以上有逻辑的正文
- **更新已有** — 为已有节点补充新的角度或修正
- **跳过** — 通用知识、琐碎事实、已覆盖、写不出"为什么"的知识点

**产出量预期**：一份素材通常产出 1~5 个节点。如果超过 5 个，暂停自检——是否有的节点其实太薄？

### 批量摄入指导

如果用户一次给了多个文件：
- **先全部通读，建立全局画面**：多个文件之间往往存在跨文件的洞见（如 A 文件的方案选择 + B 文件的问题定义 = 一个完整的 decision 节点）
- **按主题而非按文件切节点**：不要 file-1 产出 node-1、file-2 产出 node-2——应该从所有文件中抽象出 5-10 个独立的理解主题
- **宁可少写，留给 lint 补** — 批量处理时关系容易遗漏，这是正常的
- **批量摄入后建议用户跑一次 lint**

## Step 4 — 写/更新节点

每个节点 = 一个文件：`content/nodes/<id>.md`

### 文件命名
- **id = 文件名**：kebab-case 英文，如 `tcp-head-of-line-blocking.md`
- 语义化且表达观点：`subagent-trades-context-for-tokens` 而不是 `subagent-mechanism`
- 不用中文文件名（便于 URL、跨平台、git）

### Frontmatter 模板

```yaml
---
id: <与文件名一致，不含 .md>
title: <中文标题，简洁有力>
meta_type: <五选一，见下>
domains:
  - "#大类/细分"
status: <五选一，见下>
created: YYYY-MM-DD
updated: YYYY-MM-DD
wiki: <可选：物理 wiki 分组>

sources:
  - title: "来源标题"
    path: "raw/文件名.md"
    date: YYYY-MM-DD

relations:
  - to: <目标节点 id>
    type: <关系类型>
    note: "可选补充说明"

derived_from: []
splits_into: []
---
```

### 五种元类型（每个节点选一个）

| 值 | 含义 | 什么时候用 |
|---|---|---|
| `observation` | 事实 | 看到/读到/听到的原始事实，不掺自己理解 |
| `insight` | 洞见 | 自己提炼的理解、因果链、心智模型，或模型帮你发现的洞察。**最有价值的类型** |
| `decision` | 主张 | 基于理解做的选择或判断 |
| `question` | 疑问 | 还没想清的问题，挂着慢慢长 |
| `comparison` | 辨析 | X 与 Y 的对比本身值得单列 |

**判断规则**：
- "我看到了 X" → observation
- "我认为 X 是因为 Y" → insight
- "我选了 X 而不是 Y" → decision
- "X 和 Y 有什么不同" → comparison
- "我还不确定" → question

**insight 与 decision 的关键区分**：
- `insight` 是可迁移的通用理解——换一个项目、换一个场景，这个洞见依然成立。如"背压是所有生产者-消费者架构的通用问题"。
- `decision` 是特定场景下的判断——跟具体项目、具体时间点绑定，不一定能迁移。如"AI Coder 项目选用 Cursor SDK 作为 Provider"。
- 如果你发现一个"洞见"只适用于单个项目的单次决策且无法迁移到其他场景，它就是 `decision` 而非 `insight`。

### 八种关系类型

| type | 含义 | 反向展示 |
|---|---|---|
| `implements` | 我是 X 的具体实现 | 被实现 |
| `contradicts` | 我与 X 矛盾 | 被反驳 |
| `contrasts` | 我跟 X 构成有意义的对照 | 对照 |
| `extends` | 我在 X 基础上深化 | 被扩展 |
| `instance-of` | 我是 X 的一个实例 | 包含实例 |
| `prerequisite` | 理解我之前先理解 X | 后续 |
| `evolves-from` | 我从 X 的思考演化而来 | 演化为 |
| `related` | 弱关联（`[[wikilink]]` 默认值） | 相关 |

### 节点状态

| status | 含义 | 何时设 |
|---|---|---|
| `seed` | 萌芽，一两句话 | 刚提及但没展开 |
| `growing` | 在长，有内容但不完整 | 见下方详细说明 |
| `mature` | 成熟稳定 | 见下方详细说明 |
| `needs-split` | 过载 | lint 时识别 |
| `archived` | 已拆分/已废弃 | 拆分后的原节点 |

**growing vs mature 判断指南**（这是最容易模糊的区分）：

一个节点是 `mature` 当且仅当你对以下三个问题都能回答"是"：
1. **完整性**：如果半年后重读这个节点，它本身就能让你重新理解这件事，不需要去翻原始素材补充？
2. **稳定性**：你对这个理解有信心——不是"我觉得可能是这样"，而是"我确认是这样，因为我验证过/实践过/想透了"？
3. **自足性**：节点的因果链或判断逻辑写清楚了，不是只有结论没有推理？

如果任何一个回答"否"，就是 `growing`。典型的 `growing` 场景：
- 你知道结论但说不清为什么（缺机制）
- 你从代码/文章里提炼了一个模式，但还没在其他场景验证过（缺迁移）
- 你写了两段，但心里知道还有重要的角度没覆盖（缺完整性）
- 这是别人的观点你觉得有道理，但还没内化为自己的理解

`seed` 很好区分：正文就一两句话、只是个占位符。

### 正文写作规范

**写理解，不写摘要。写因果链，不写功能描述。**

#### 坏例子 vs 好例子

| 坏（摘要/搬运） | 好（理解） |
|---|---|
| "Subagent 有独立上下文，不传 messages" | "Subagent 本质上是拿信息损失换 token 效率——主 agent 只传一句 description 给子 agent，这意味着子 agent 无法理解上下文中的隐含意图。这个 trade-off 在简单任务中划算（grep 文件不需要知道用户聊了什么），在需要判断力的任务中会出错。" |
| "Redis Stream 支持消费者组和消息持久化" | "选 Redis Stream 而非 Pub/Sub 做事件总线，本质是从'通知即遗忘'升级到'通知可回溯'——SSE 断线重连时消费者可以从 lastEventId 继续消费，这消除了传统 SSE 最大的工程痛点。" |
| "AG-UI 用 JSON Patch 做增量更新" | "AG-UI 选 JSON Patch 而非全量 state 推送，背后的工程约束是：token 流式生成时 state 每秒变几十次，全量推每次都包含未变部分，带宽和渲染成本都是 O(state_size × update_freq)。Patch 把成本降到 O(delta)，但代价是客户端必须维护本地 state，且 patch 顺序敏感——一旦丢包就要全量恢复。" |

#### insight 节点的三段式结构

1. **现象/矛盾**（1-2 句）：观察到了什么，什么让你觉得"这里有东西"
2. **机制/因果**（主体，3-8 句）：为什么会这样？因果链是什么？设计者在什么约束下做了什么选择？代价是什么？
3. **迁移/启示**（1-2 句）：这个理解在其他场景怎么用？它是否是某个更通用模式的实例？

**如果你写不出第 2 段（机制），说明这个知识点还不够格成为 insight——降级为 observation 或直接跳过。**

#### observation 节点的写法

observation 不需要三段式，但需要：
- 准确描述"你看到了什么"
- 标注来源和可信度
- 适当长度：3-8 句话
- 不要试图解释 why（那是 insight 的事）

#### 其他规则
- 用 `[[node-id]]` 引用其他节点，这些会自动成为 `related` 关系
- 长度保持 3 行 ~ 3 屏。更长就该拆
- 正文用中文
- **禁止 bullet point 列表作为正文主体** — 如果你的正文是一组要点列表，你在做笔记不是在写理解。把列表转化为有逻辑连接的段落。

## Step 5 — 自检关卡（写完节点后、建关系前）

对每个刚写完的节点，做以下检查。**任何一项不通过就修改或删除该节点：**

| 检查项 | 不通过的表现 | 修复方式 |
|---|---|---|
| 标题有张力吗？ | 标题只是概念名（"MCP 三层架构"） | 改为表达洞察（"MCP 三层架构让企业集成从写代码变成配 JSON"） |
| 正文有因果吗？ | 正文在"介绍"而非"解释" | 补上 why/trade-off，或降级为 observation |
| meta_type 准确吗？ | 标了 insight 但正文只描述 what | 降为 observation |
| 跟已有节点重复吗？ | 跟已有节点 >70% 重叠 | 合并到已有节点 |
| 太薄了吗？ | 正文 < 100 字且不是 seed | 要么写深，要么标 seed |

## Step 6 — 建立关系（最重要的步骤）

**如果只能做好一件事，就做好关系发现。** 孤立的节点只是笔记，连接起来才是知识。

对每个新建/更新的节点，执行以下关系发现检查：

1. **同域扫描**：浏览所有同 domain 节点的标题，找语义相近的——可能是 extends / prerequisite / contrasts
2. **跨域类比**：浏览不同 domain 的节点，找"结构相似但领域不同"的——如"延迟满足"同时连接育儿和产品管理。**跨域关系是 wiki 最有价值的产物**
3. **回答疑问**：检查是否有 question 节点被这个新理解部分回答了——如果是，建立关系并考虑更新 question 的 status
4. **升级事实**：检查是否有 observation 节点现在可以被赋予因果理解（升级为 insight）——因为新素材提供了"为什么"
5. **补充对照**：如果新节点是"A 方案的选择"，检查是否存在"B 方案"节点，应建 contrasts 关系

在 frontmatter 的 `relations` 中声明关系，同时在正文中自然地使用 `[[node-id]]`。

## Step 7 — 更新索引

在 `content/meta/index.md` 中添加/更新条目：
```markdown
- [[node-id]] — 一句话描述
```
按 domain 或 meta_type 分组，保持索引可扫描。

## Step 8 — 记录日志

在 `content/meta/log.md` 末尾追加：
```markdown
## [YYYY-MM-DD] ingest | <素材标题>
- 来源: raw/<文件名>
- 新建: node-a, node-b, node-c
- 更新: node-d
- 关系: node-a →extends→ node-x, node-b →contrasts→ node-y
```

## Step 9 — 向用户汇报

告诉用户：
1. 处理了什么素材
2. 新建了哪些节点（id + title + meta_type）
3. 更新了哪些节点
4. **主动跳过了什么**（简述为什么跳过——"通用知识/已覆盖/太薄写不出 why"）
5. 建立了哪些关系（重点说跨域关系）
6. 有无不确定的地方（请用户确认）
7. 建议人工抽检的节点（≤ 3 个，说明为什么建议抽检）

**汇报格式示例**：

```
摄入完成！从 ai-coder 笔记（13 文件）中提炼出 7 个节点：

新建节点：
| id | title | type | status |
|---|---|---|---|
| subagent-context-isolation-tradeoff | Subagent 用信息损失换 token 效率 | insight | growing |
| middleware-as-capability-config | 洋葱模型让能力扩展变成声明式配置 | insight | mature |
| sse-reconnect-needs-persistent-stream | SSE 断线重连的前提是事件流可回溯 | insight | mature |
| ai-coder-cursor-positioning | AI Coder 选择对标 Cursor 而非 Copilot | decision | growing |
...

跳过内容：
- NestJS 生命周期、JWT/Guard 机制 → 通用知识，不建节点
- Redis 基础数据结构 → 通用知识
- 前端组件细节 → 实现细节，无可迁移洞见

关键关系：
- subagent-context-isolation-tradeoff ↔contrasts↔ summarization-as-lossy-compression
  （都在解决上下文膨胀，但一个是空间隔离，一个是时间压缩）

建议抽检：ai-coder-cursor-positioning（decision 类容易过时）
```

## 领域标签（domains）约定

格式：`#大类/细分`。

**标签表示"这个理解在哪些领域可迁移"，不是"素材来自哪个项目"。** 一个从 AI Coder 项目中提炼出的通用工程原则（如背压机制），应该同时打 `#技术/工程`，而不只是 `#工作/AI编码助手`。

当前已用：

- `#技术/前端`、`#技术/网络`、`#技术/工程`、`#技术/AI`
- `#工作/AI编码助手`、`#工作/职业`
- `#生活/健康`、`#生活/日记`
- `#自我管理`
- `#思考`

可随时增加新标签。一个节点可打多个标签。

## 红线

- ❌ 不修改 `content/raw/` 下的任何文件
- ❌ 不删除节点文件（只能 archive）
- ❌ 不删除 log 历史条目
- ❌ 不写空洞的搬运——正文必须是理解，不是原文复制
- ❌ 不写孤立节点——每个新节点至少找到一条关系
- ✅ 写之前先检查已有节点，避免重复
- ✅ id 语义化、title 用中文
- ✅ status 诚实——两句话的节点就是 seed，不要标 mature
