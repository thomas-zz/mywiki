# myWiki 产品需求文档（PRD）

## 1. 产品定位

myWiki 是一个**个人知识理解系统**。它不是博客（给别人看的橱窗），而是给自己看的镜子——帮助使用者看见自己的思考形状、发现知识之间的联系、让理解随时间自然进化。

核心差异点：
- 最小单元不是"文章"，是"一个理解"（节点）
- 知识以图（graph）而非树（directory）组织
- LLM 全权维护，用户只管输入素材和审阅
- 节点会自我进化（拆分/合并/重连），不是写完就不动
- 未完成的思考、疑问、半成品理解都允许存在且可见

## 2. 目标用户

第一阶段：仅自己使用（单用户）。部署在公网，简单密码保护。

## 3. 核心理念

> 简单知识不再稀缺，AI 能背 API、查事实、写样板代码。学习的重心从机械记忆转向系统理解、问题定义、质量判断、因果关系和领域直觉。myWiki 把文章、资料、事实重新投影为节点和关系，因为不同的信息组织方式帮助人获得洞见。

三条设计公理：
1. **涌现优于分类**——不预设分类树，让 hub 自己浮出水面
2. **理解优于记忆**——事实类节点做轻（锚点），洞见类做重（主体）
3. **自进化是一等公民**——节点有生命周期，拆分/合并是正常操作

## 4. 功能需求

### 4.1 节点系统

#### 4.1.1 节点定义
- 每个节点 = 一个 markdown 文件 = 一个"理解的最小单元"
- 长度 3 行 ~ 3 屏不等
- 所有节点平铺存放，不按目录分类

#### 4.1.2 五种通用元类型（meta_type）

| 元类型 | 标签 | 含义 | 适用范围 |
|---|---|---|---|
| `observation` | 👁 事实 | 看到/听到/读到的，不掺自己的理解 | 全领域 |
| `insight` | 💡 洞见 | 自己提炼的理解、心智模型、因果链 | 全领域 |
| `decision` | 🎯 主张 | 基于理解我选择了什么、相信什么 | 全领域 |
| `question` | ❓ 疑问 | 还没想清的问题，挂着慢慢长 | 全领域 |
| `comparison` | ⚖️ 辨析 | X 与 Y 的对比、取舍本身值得单列 | 全领域 |

#### 4.1.3 领域标签（domains）
- 自由打标，不强制层级
- 格式：`#大类/细分`，如 `#技术/网络`、`#生活/育儿`、`#工作/职业`、`#思考`
- 一个节点可打多个标签——跨域 hub 自然涌现

#### 4.1.4 洞察来源（insight_origin）

`insight` 节点额外标注来源维度，区分两种知识来源：

- `explicit`：用户或素材作者已经显式写出的洞察，LLM 要尊重原意记录
- `inferred`：LLM 基于 wiki 中的事实节点和关系自行推导出的洞察，是 wiki 自进化的主要来源
- `mixed`：两者混合，暂时无法合理拆开

这个字段帮助系统区分“忠实记录”与“自进化推断”，避免把素材原意和 wiki 推理混写在一起。

#### 4.1.5 节点状态（status）与生命周期

```
seed（萌芽）→ growing（生长中）→ mature（成熟）→ needs-split（过载）→ 拆分
                                                                    ↓
                                                               archived（归档/索引化）
```

#### 4.1.6 语义关系（relations）

每个节点可声明指向其他节点的带语义关系：

| 关系类型 | 含义 | 反向展示标签 |
|---|---|---|
| `implements` | 我是 X 的具体实现 | 被实现 |
| `contradicts` | 我反驳/与 X 矛盾 | 被反驳 |
| `contrasts` | 我跟 X 是有意义的对照 | 对照 |
| `extends` | 我在 X 之上扩展 | 被扩展 |
| `instance-of` | 我是 X 的一个实例 | 包含实例 |
| `prerequisite` | 理解我之前先理解 X | 后续 |
| `evolves-from` | 我是 X 思考的演化版 | 演化为 |
| `related` | 弱关联（wikilink 默认） | 相关 |

- 关系可附 note 说明
- 反向边在构建时自动生成
- 正文 `[[node-id]]` wikilink 自动解析为 `related` 类型关系（若未显式声明）

#### 4.1.7 自进化机制

- 节点 frontmatter 含 `derived_from` / `splits_into` 字段追溯演化历史
- LLM lint 时识别"过载信号"：正文过长 + 入链多样性高 + 子主题密度大
- 拆分操作流程：LLM 提议 → 用户确认 → 拆分 → 重路由引用 → 留 derived_from 痕迹
- 网站展示"最近被拆分/合并的节点"

#### 4.1.8 来源（sources）
- 每个节点可关联 0~N 个来源
- 来源指向 `raw/` 目录下的原始素材文件
- 来源有 title、path、date 字段

### 4.2 网站视图

#### 4.2.1 首页
- **不是文章列表**，是"你脑子的形状"
- 展示内容：
  - 入口 hub（最多被引用的节点）
  - 跨领域节点（多 domain 标签的节点）
  - 开放问题（status=question 的节点）
  - 正在生长的节点（seed / growing）
- 元类型图例常驻

#### 4.2.2 节点详情页
- 正文（markdown 渲染，wikilink 可点击可悬停预览）
- 来源区（素材出处列表）
- 语义关联区（分"我指向"和"指向我"两组，每条含关系类型 chip + 节点标题 + 元类型 + note）
- 局部图谱（当前节点 + 1 跳邻居，中心放大，concentric 布局）
- 面包屑 + 返回按钮

#### 4.2.3 全局图谱
- 所有节点 + 所有关系的力导图
- 节点大小 = 入链数（hub 自然大）
- 节点颜色 = 元类型
- 可缩放/拖拽
- 点击节点跳转详情

#### 4.2.4 涌现 · 自进化页
- hub 节点排行（按入链）
- 跨域 hub（多 domain + 高入链）
- 开放问题积压（按存在天数排序）
- 候选拆分（overload signal：正文长 + 入链多）
- 最近更新
- 孤岛节点（无入无出）

#### 4.2.5 领域页
- 按 domain 标签筛选所有节点
- 展示节点列表 + 元类型标记 + 正文摘要

#### 4.2.6 全部节点页
- 按元类型分组展示
- 每组含计数、描述

#### 4.2.7 搜索
- 全文搜索（标题 + 正文 + 领域标签）
- 标题匹配优先排序
- 结果含高亮片段
- 侧边栏常驻搜索入口

### 4.3 悬停预览

- 鼠标悬停在任何节点链接上（关联区、正文 wikilink、列表项）→ 弹出浮层
- 浮层含：标题、元类型、状态、入/出度、领域标签、正文前 280 字
- 不需要点击跳转就能快速浏览
- 图谱区域**不**触发弹层（避免遮挡和点击冲突）

### 4.4 导航与路由

- 所有视图含 `← 返回` 按钮 + 面包屑
- 浏览器前进/后退正常工作
- 侧边栏含：视图导航、入口 hub 快捷链接、领域标签索引
- URL 以 hash 路由为基础（静态部署友好）

### 4.5 认证

- 简单密码保护（第一阶段）
- 登录状态 session 内保持
- 部署时可升级为 Cloudflare Access / Vercel password protection

### 4.6 数据流水线（写入侧）

用户本地流程：
1. 用户在 Obsidian（或任意编辑器）写笔记/粘贴素材
2. LLM（Claude Code）在本地执行 ingest：读素材 → 写/更新节点 markdown → 更新 index
3. `git push` 到 GitHub

GitHub 流水线：
4. GitHub Action 触发
5. 解析所有节点 markdown（gray-matter 提取 frontmatter + remark 处理正文）
6. 生成 `data.json`（节点列表 + 关系图 + 指标）
7. 构建静态网站
8. 部署到 Vercel / Cloudflare Pages / GitHub Pages

### 4.7 跨 wiki 策略

- 物理上按 wiki 分目录（便于 LLM 维护、便于隐私管控）
- 逻辑上**允许跨 wiki 引用**——网站上节点不分筒仓
- 私密 wiki 的节点对未认证用户不可见
- `wiki` 字段仅作 metadata / 筛选用

## 5. 非功能需求

### 5.1 性能
- 纯静态部署，CDN 加速
- 首屏加载 < 2s（200 节点规模）
- 图谱渲染：100 节点以内实时流畅，500 节点可接受卡顿但可用

### 5.2 可维护性
- 数据源是纯 markdown（git 版本管理）
- 网站只读，不回写 git
- 新增节点只需写一个 markdown + push，无需改代码

### 5.3 离线 / 本地使用
- 网站支持完全本地运行（`pnpm dev` 或静态文件直接打开）
- 用户可在设置中选择本地目录作为 wiki 仓库，支持修改路径
- 无网络环境下仍可浏览全部内容、搜索、查看图谱
- 本地模式 = 线上模式的超集（线上只能看，本地还能触发构建）

### 5.4 LLM 本地维护流程（Skill 驱动）
- 提供一份标准 Skill 文件，告诉任何 LLM Agent 如何解析素材并写入 wiki
- 用户的工作流：① 扔素材给 LLM → ② LLM 按 Skill 自动写节点 → ③ 用户 review → ④ git push → ⑤ 网站更新
- Skill 是这个系统的"编程接口"——换 LLM 不影响 wiki 结构
- 不做服务端 Agent（第一阶段全靠用户本地手动触发）

### 5.5 可扩展性（后续版本考虑）
- LLM 查询接口（网页内提问）
- 移动端适配
- RSS 输出
- 节点版本历史（git blame 可视化）
- 定时 lint 报告
- 服务端 Agent（GitHub webhook 自动 ingest）

## 6. 不做的事（明确排除）

- 不做多用户 / 协作 / 评论
- 不做 CMS 后台 / 在线编辑（编辑在本地 Obsidian + LLM 完成）
- 不做传统博客功能（时间线、RSS 聚合、评论区）
- 不做移动端 App（第一阶段 web only）
- 不做搜索引擎 SEO 优化（个人工具，不求曝光）
